# RK3568 五口并发 UDP 网络性能测试报告

## 1. 测试目的
验证 RK3568 Ubuntu 系统板在 **1 个光口（eth0）+ 4 个电口（eth2/eth3/eth4/eth5）** 并发 UDP 打流场景下的网络吞吐、丢包与系统资源稳定性。

## 2. 测试依据与数据来源
- 测试要求与拓扑说明：`readme.md`。
- 测试脚本：`start_test.sh`、`stop_test.sh`。
- 网络吞吐/丢包日志：`iperf3_server_eth*.log`。
- 实时网口速率：`iface_rates.csv`。
- 系统压力与网卡统计（由 nmon 转换）：`rk3568_stress.csv`。
- sar 网络统计：`sar_network.log`。

> 说明：根据任务要求，仅解析了 `rk3568_stress.csv`（未重复解析 `.nmon` 原始文件）。

## 3. 测试配置与方法核对

### 3.1 脚本配置核对
`start_test.sh` 中服务端监听配置如下：
- eth0: `192.168.8.200:5201`
- eth2: `192.168.2.200:5202`
- eth3: `192.168.3.200:5203`
- eth4: `192.168.4.200:5204`
- eth5: `192.168.5.200:5205`

并同时开启：
- `iperf3` 服务端日志采集；
- `sar -n DEV 1 600`；
- 1 秒粒度接口速率 CSV；
- `nmon -f -s 1 -c 600`。

### 3.2 与 README 的一致性说明
- README 中文字命令示例写为统一 `-p 5202`，但脚本实际对 5 个接口分别使用了 5201~5205 独立端口。实测日志也确认 5 个端口均已建立连接并有独立数据流。

## 4. 测试时长与数据完整性
- 期望时长：600 秒（脚本配置）。
- 实测 `iperf3` 总结区间约：`0.00-650.xx sec`（各口约 650 秒）。
- `iface_rates.csv` 有效采样：580 行。
- `rk3568_stress.csv` 中 CPU/MEM/NET 各 600 个时间片，`ZZZZ` 时间戳 600 条。

结论：数据整体完整，但存在“实际测试时长 > 预设 600 秒”的现象（约 +50 秒），可能与人工停止时机、并发进程结束顺序或收尾阶段缓冲刷新有关。

## 5. 结果汇总

## 5.1 iperf3 最终接收统计（按接口）
以下为各日志中末尾 `[SUM] ... receiver` 汇总值：

| 接口 | 接收吞吐(Mbps) | 抖动(ms) | 丢包率 | 丢包/总包 |
|---|---:|---:|---:|---:|
| eth0 | 105 | 0.035 | 87% | 38,658,668 / 44,520,775 |
| eth2 | 124 | 7.792 | 81% | 5,250,626 / 6,481,694 |
| eth3 | 253 | 0.066 | 54% | 16,228,393 / 30,327,461 |
| eth4 | 236 | 0.267 | 70% | 31,377,501 / 44,520,647 |
| eth5 | 228 | 0.121 | 72% | 32,106,767 / 44,885,829 |
| **合计** | **946** | - | - | - |

**观察：**
- 五口并发总吞吐约 **946 Mbps**。
- 但 UDP 丢包率非常高（54%~87%），链路质量未达到“低丢包压力测试”预期。

## 5.2 iface_rates.csv（网口瞬时速率）
对 `iface_rates.csv` 中每口 `rx_MBps` 进行统计（仅将 >1 MB/s 视为有效打流样本）：

| 接口 | 有效样本数 | 平均接收(MB/s) | P95(MB/s) | 峰值(MB/s) |
|---|---:|---:|---:|---:|
| eth0 | 454 | 46.44 | 72.91 | 108.12 |
| eth2 | 454 | 114.67 | 162.98 | 275.91 |
| eth3 | 454 | 95.85 | 135.34 | 253.00 |
| eth4 | 454 | 140.72 | 204.19 | 391.49 |
| eth5 | 454 | 95.12 | 142.95 | 226.07 |

**观察：**
- 瞬时峰值有较大抖动，尤其 eth4/eth2 存在明显突增。
- `tx_MBps` 长期接近 0，符合本测试场景（板端主要作为 iperf3 server 接收 UDP 流）。

## 5.3 sar 平均值（网络利用率）
`sar_network.log` 的 `Average:` 区段显示：
- eth0 平均 `rxkB/s` ≈ 32199.10，`%ifutil` ≈ 26.38
- eth2 平均 `rxkB/s` ≈ 81543.31，`%ifutil` ≈ 66.80
- eth3 平均 `rxkB/s` ≈ 67968.97，`%ifutil` ≈ 55.68
- eth4 平均 `rxkB/s` ≈ 99210.89，`%ifutil` ≈ 81.27
- eth5 平均 `rxkB/s` ≈ 67517.02，`%ifutil` ≈ 55.31

**观察：**
- eth4 平均利用率最高（81.27%），eth0 最低（26.38%）。
- 与 iperf 最终吞吐排序基本一致（eth3/eth4/eth5 高于 eth0/eth2）。

## 5.4 系统资源（rk3568_stress.csv）
- CPU（CPU_ALL：User%+Sys%）
  - 平均忙碌度：**96.73%**
  - P95：**100.00%**
  - 峰值：**100.00%**
- 内存（MEM：memfree）
  - 最小空闲：**6093.3 MB**
  - 平均空闲：**6099.8 MB**
  - 最大空闲：**6108.5 MB**

**观察：**
- CPU 在绝大部分时间处于高负载甚至打满状态，属于本次高丢包的重要风险点。
- 内存资源充足且波动很小，不构成瓶颈。

## 6. 结论
1. **吞吐角度**：五口并发 UDP 汇总吞吐接近 1Gbps（约 946Mbps），说明系统具备较高聚合接收能力。  
2. **质量角度**：丢包率显著偏高（54%~87%），当前配置下无法满足低丢包稳定传输目标。  
3. **瓶颈判断**：结合 CPU 长时间接近 100%，推测主要瓶颈在 CPU 处理能力/中断与协议栈处理路径，而非内存容量。  
4. **端口差异**：eth0、eth2 表现弱于 eth3/eth4/eth5，建议重点核查驱动参数、IRQ 绑核与交换/PHY 链路状态一致性。  

## 7. 优化建议
1. **先降负验证**：将单口或总负载从 200M×4 适当下调，绘制“负载-丢包率”曲线，确定拐点。  
2. **CPU/IRQ 优化**：检查网卡中断亲和性（`/proc/interrupts`、`smp_affinity`），将高流量口 IRQ 分散到不同 CPU 核。  
3. **网卡队列与缓存**：调优 ring buffer、backlog、rmem/wmem（`ethtool -g/-G`、`sysctl`）并对比前后丢包。  
4. **分口复测**：分别单口/双口/五口逐级并发，定位是总线瓶颈还是特定网口路径问题。  
5. **时长控制**：优化停止流程，避免 600 秒任务延长到 650 秒，保证不同批次数据可比性。  

## 8. 附录：本报告使用的解析口径
- iperf3：使用每个 `iperf3_server_eth*.log` 末尾 `[SUM] ... receiver` 作为该口最终结果。  
- 速率统计：`iface_rates.csv` 中 `rx_MBps > 1` 视为有效流量样本，计算均值/P95/峰值。  
- 系统统计：仅解析 `rk3568_stress.csv`（CPU_ALL、MEM、NET）。
